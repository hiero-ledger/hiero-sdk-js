@hiero-ledger/proto full vs minimal

Compares the legacy pbjs-generated package with the new tree-shakeable ts-proto build.

Why is minimal so much smaller?

Four mechanisms stack on top of each other. The first does most of the work.

1

ESM tree-shaking beats the CJS monolith

The full package is one CJS file with all 377 Hedera message classes bolted onto a single exports.proto object. Bundlers can’t statically prove which classes you actually touch, so they keep every one.

The minimal package emits 92 separate ESM modules, one per proto. Importing FileAppendTransactionBody pulls in ~3 files. Rollup drops the rest at the linker stage.

Accounts for roughly 85–90% of the raw saving on its own.
2

Per-transaction proto scoping

Even within the messages a bundler keeps, scope matters. The full basic_types.proto declares 72 messages — from AccountID to NodeAddressBook — all co-located.

Minimal’s file_append.proto declares one message and reaches into basic_types.proto only for the types it references (just FileID). No orphan siblings travel along.

3

Leaner runtime library

Before a single message is defined, the two packages already differ by ~45 KB at the runtime layer:

Full (pbjs) Minimal (ts-proto)
protobufjs/minimal ≈ 30 KB @bufbuild/protobuf/wire ≈ 5 KB
Long.js ≈ 20 KB native bigint (0 KB)
JSON + reflection + verify binary wire only
4

Codegen drops everything optional

Each flag below removes a chunk of boilerplate from every generated message. Multiply by 377 messages and it adds up fast.

  • outputJsonMethods=false — no toJSON/fromJSON
  • outputSchema=false — no runtime schema
  • outputServices=false — no gRPC stubs
  • forceLong=bigint — no Long.js dependency

The full-envelope and body-only bundles from the full package weigh the same — pbjs is a monolith, you pay for all 377 messages either way. Minimal’s envelope costs only ~22 KB more than the body alone, because the per-transaction TransactionBody has a single-case oneof data that closes the graph.

Why is the gzipped saving smaller than the raw saving?

Gzip loves repetition. The full package has 377 near- identical encoder/decoder pairs, so DEFLATE compresses it ~9× (688 KB → 78 KB). Minimal is already lean and less repetitive, so it only compresses ~3× (15.5 KB → 4.8 KB). The gap narrows after gzip, but minimal stays vastly smaller on the wire.
